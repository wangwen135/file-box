## 1. 外部配置文件 `application.yml` 架构问题

> 状态：🔵 待处理  
> 优先级：中

### 1.1 现状

项目里存在**两份同名**的 `application.yml`，职责不同、读取方不同：

| 文件 | 内容 | 读取方 | 是否打入 jar |
|------|------|--------|------|
| **根目录** `application.yml` | 业务配置：用户账号、存储空间等 | `ConfigService` 用**手写 YAML 解析器**读取（`new File("application.yml")`，相对工作目录），**不走 Spring** | 否（外部文件） |
| `src/main/resources/application.yml` | Spring 框架配置：`server.port`、`spring.servlet.multipart` | **Spring Boot** 自动加载（classpath） | 是 |

**为什么业务配置在根目录、而不是 `src` 下**：后台管理界面"保存"时，`ConfigService.saveConfig()`（`ConfigService.java` 的 `saveConfig` 方法，约 232 行）会**覆盖写回**这个外部文件。它必须是运行时可写的外部文件——放进 `src/main/resources`（打入 jar）后打包就无法改写了。因此这是**有意设计**，`README` 的发布包结构（`file-box.jar` + 外部 `application.yml` + 启动脚本）也印证了这一点。

### 1.2 已识别问题

1. **同名混淆**：两份 `application.yml` 仅靠位置区分职责（`src` 版=框架配置、根目录版=业务配置），容易搞混，新人难以判断该改哪份。
2. **敏感信息入了 git**：根目录 `application.yml` 含**用户密码哈希**（BCrypt），却被 git 追踪——`.gitignore` 未忽略它。密码哈希不应进版本库。
3. **打包逻辑错位**：`src/assembly/assembly.xml` 打包的是 `src/main/resources/application.yml`（仅框架配置），但运行时 `ConfigService` 读的是工作目录下的外部版本（业务配置）。结果发布包里的 `application.yml` 只有框架配置，**部署后需要手动填入业务配置**（用户、存储空间）才能正常工作，与"开箱即用"的预期不符。

### 1.3 候选整改方案（择期实施，不在本次执行）

- **方案 A — 重命名外部业务配置**：把外部业务配置改名为 `filebox.yml`（或 `config.yml`），与 Spring 的 `application.yml` 彻底区分，消除同名混淆。
  - 涉及：`ConfigService.CONFIG_FILE` 常量、`README` 发布包结构说明、`src/assembly/assembly.xml`（改为打包模板/示例文件）、迁移现有根目录 `application.yml` → `filebox.yml`。
- **方案 B — 安全处理**：保持文件名不变，但将含密码哈希的外部配置加入 `.gitignore`，并新增 `application.example.yml`（或 `filebox.example.yml`）模板供参考与初始化。
- **方案 C — A + B 同时做**：既重命名消除混淆，又处理敏感信息入库问题。**推荐**。

> 注意：业务配置仍需保持"外部可写"（不能合并进 `src` 下用 Spring `@ConfigurationProperties` 管理），否则后台"保存"写回机制（`saveConfig`）将失效。

### 1.4 相关文件
